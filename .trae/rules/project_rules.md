## 数据存储边界
- ✅ **历史行情数据**：存储A股历史K线数据（日线级别）
- ❌ **实时行情数据**：不存储实时行情，仅通过API实时查询
- ✅ **数据更新频率**：每日收盘后更新一次历史数据
- ✅ **数据范围**：最近3年历史K线数据

## 数据采集规范

### 交易时段禁止采集规则

**核心原则**：交易日盘中（9:30-15:00）**禁止采集当日K线数据**

| 时段 | 可采集？ | 说明 |
|------|----------|------|
| 交易日 9:30-15:00 | ❌ 禁止 | 盘中行情不完整，采集会导致数据污染 |
| 交易日 15:00 后 | ✅ 允许 | 正常采集当日数据 |
| 非交易日 | ✅ 允许 | 可采集历史数据 |

**实现要求**：
1. `data_collect.py` 必须调用 `_check_market_closed()` 检查市场状态
2. 非收盘后时段执行时，脚本必须**主动退出**并输出提示
3. 仅当 `--date` 参数明确指定历史日期时，才允许采集历史数据

**例外情况**：
- `--date YYYY-MM-DD` 参数可强制指定历史日期（用于补采）
- 初始化全量历史数据时不受此限制

## 数据质量规范

### 退市股票过滤规则

**核心原则**：选股流程中**必须排除已退市/数据过旧的股票**

| 检查点 | 要求 | 实现方式 |
|--------|------|----------|
| 数据新鲜度 | 最新数据日期必须在30天内 | `data_freshness_filter` |
| 退市名称检测 | 名称含"退市"关键词 | `delisting_filter` |
| ST风险股检测 | 名称含"ST"关键词 | `st_filter` |
| 停牌检测 | 成交量>0为必要条件 | `suspension_filter` |

**配置位置**：
- 过滤器配置：`config/filters/fund_behavior_filters.yaml`
- 过滤器实现：`filters/market_filter.py` → `DataFreshnessFilter`

**强制要求**：
1. 所有选股流程（包括主力痕迹共振扫描）必须在输出前应用数据新鲜度检查
2. `scan_mainforce_signals()` 函数新增 `max_data_age_days` 参数，默认30天
3. 数据过旧的股票不得出现在任何推荐报告中

**违规风险**：已退市股票被推荐给用户 → 致命问题

## 四、 技术栈与工具链 (Tech Stack)

* **后端**：Python 3.11+ / Flask / SQLAlchemy (需兼容 MySQL 5.0+)
* **前端**：Next.js 16+ / TypeScript / Tailwind CSS
* **存储**：MySQL 5.0+ / Redis
* **部署**：Docker / Docker Compose

### 数据处理技术栈

* **数据格式**：Parquet - 高效列式存储格式
  - 压缩率高，查询速度快
  - 支持列式读取，适合大规模数据分析
  - 文件大小：平均20KB/股票，总计约120MB（5000+股票）

* **数据处理库**：
  - **Polars** - 高性能DataFrame库
    - 比Pandas快10-100倍
    - 内存效率高，支持懒加载
    - 支持多线程和SIMD优化
    - 适用场景：大规模数据清洗、转换、聚合

  - **DuckDB** - 嵌入式分析数据库
    - 支持SQL查询Parquet文件
    - 零配置，无需服务器
    - 支持窗口函数、聚合、JOIN等复杂查询
    - 适用场景：复杂SQL分析、多表关联、数据挖掘

* **数据存储结构**：
  ```
  data/
  ├── kline/                    # K线数据目录
  │   ├── 000001.parquet       # 每个股票一个文件
  │   ├── 000002.parquet
  │   └── ...
  ├── stock_list.parquet       # 股票列表
  └── stock_list.csv           # 股票列表CSV格式
  ```

* **数据字段**：
  - code: 股票代码
  - trade_date: 交易日期
  - open: 开盘价
  - close: 收盘价
  - high: 最高价
  - low: 最低价
  - volume: 成交量

* **性能优化建议**：
  - 使用Polars进行数据清洗和转换
  - 使用DuckDB进行复杂SQL查询
  - 使用Parquet格式存储，减少I/O时间
  - 批量处理数据，避免逐条查询
