### 5. 存储性能优化

**当前问题**: 5394个独立Parquet文件，文件系统扫描慢，文件描述符压力大

**优化方案**:

| 方案 | 描述 | 适用场景 |
|------|------|----------|
| 按月分区 | 按交易月份合并数据文件 | 历史数据查询 |
| 按首字母 | 按股票代码首字母分区 | 批量处理 |
| DuckDB中间层 | 使用DuckDB处理小文件 | 复杂SQL分析 |

**DuckDB性能优势**:
```sql
-- 直接SQL查询，无需加载全部文件
SELECT code, close FROM 'data/kline/*.parquet' 
WHERE trade_date = '2026-04-01';
```

## 问题诊断与解决方案

### 问题1: 盘中采集数据导致数据不完整

**问题描述**: 盘中（9:30-15:00）采集的K线数据不是最终数据，是实时波动数据

**根本原因**: 
1. cron 可能在 15:30 之前触发
2. 数据源 `web.ifzq.gtimg.cn` 返回的是包含当日实时数据的历史K线
3. 盘中数据会随价格波动，不稳定

**解决方案**:
1. ✅ 代码层：`data_collect.py` 已实现 `_check_market_closed()` 检查
2. ✅ 配置层：cron 任务设置为 15:30 执行
3. ⚠️ **关键修复**：增加数据"最终确认"机制，标记数据是否为"收盘后确认数据"

### 问题2: 盘后分析数据不对齐

**问题描述**: 各任务使用的数据日期不一致

**根本原因**:
- `market_review`、`picks_review`、`precompute` 等任务可能使用不同时间点的数据
- 没有统一的数据版本控制

**解决方案**:
1. 质检通过后，锁定"今日数据版本"
2. 后续任务使用质检后的数据快照
3. Redis 记录数据版本号

## 系统架构优化

### 1. 触发机制改进：从时间触发到信号触发

**当前问题**: 基于 Wall Clock Time（墙上时间），任务可能基于"半成品数据"运行

**改进方案**:
```
┌─────────────────────────────────────────────────────────────────────────┐
│                        信号触发机制                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  15:30 data_fetch ──成功──► Redis: data:ready=true                   │
│                                      │                                  │
│                                      ▼                                  │
│  17:00 data_quality_check ◄──检测──信号                                 │
│       │                                                                  │
│       │──通过──► Redis: data:version:YYYYMMDD (locked)                 │
│       │                  │                                              │
│       │                  ▼                                              │
│       │     后续任务读取锁定版本，不读 data/kline/                      │
│       │                                                                  │
│       └──失败──► 触发 data_fetch_retry (17:10)                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2. 数据快照机制：读写分离

**当前问题**: 补全过程中数据可能被正在运行的分析任务读取

**改进方案**:
```
data/
├── kline/                    # 写入目录（采集脚本使用）
│   └── *.parquet
├── snapshot/                 # 快照目录（分析任务使用）
│   └── 20260401/
│       └── *.parquet
└── index/                    # 指数数据
```

### 3. 分级质检标准

| 股票类别 | 完整率要求 | 缺失容忍 | 告警级别 |
|----------|------------|----------|----------|
| 沪深300成分股 | ≥99% | ≤1只 | CRITICAL |
| 中证500成分股 | ≥99% | ≤5只 | CRITICAL |
| 全市场 | ≥90% | ≤500只 | WARNING |
| 连续缺失 | ≤2天 | 触发CRITICAL | - |

### 4. 过滤机制增强

**新增过滤规则**:
- 次新股过滤：上市未满60天
- 停牌股过滤：当日无成交
- 流动性过滤：成交额<1000万
- ST股过滤：股票名称含"ST"
