# Kestra 集成需求对齐文档

## 项目背景

XCNStock 是一个 A 股量化交易系统，目前使用本地工作流框架执行任务。为了提升可观测性、调度可靠性和运维效率，需要与 Kestra 工作流平台深度集成。

## 现状分析

### 已有基础
- ✅ Kestra API 配置已存在 (`.env`)
- ✅ 基础部署脚本 (`kestra/deploy.py`)
- ✅ 2 个工作流定义 (数据流水线、盘前报告)
- ✅ 本地工作流框架 (`core/workflow_framework.py`)
- ✅ 丰富的 pipeline 脚本 (`scripts/pipeline/`)

### 当前痛点
1. **部署流程不完整** - 缺少自动化部署机制
2. **监控能力不足** - 无法实时查看执行状态和日志
3. **错误处理薄弱** - 失败通知和自动恢复机制不完善
4. **缺少动态参数** - 工作流参数硬编码，不够灵活

## 集成目标

### 核心目标
1. **完整部署体系** - 一键部署所有工作流到 Kestra
2. **统一监控面板** - 通过 Kestra UI 监控所有任务
3. **可靠错误处理** - 失败自动重试、告警通知
4. **灵活参数传递** - 支持运行时动态参数

### 边界界定
- ✅ **范围内**：工作流定义、部署脚本、监控集成、错误通知
- ❌ **范围外**：Kestra 服务器部署、底层基础设施

## 歧义澄清

### 问题 1：工作流触发方式
**选项 A**：完全由 Kestra Schedule 触发（推荐）
**选项 B**：保留本地 cron，Kestra 仅做监控

**建议**：选项 A，利用 Kestra 的原生调度能力

### 问题 2：失败处理策略
**选项 A**：Kestra 层重试 + 告警
**选项 B**：应用层重试 + Kestra 告警
**选项 C**：双层重试机制

**建议**：选项 C，Kestra 配置 3 次重试，应用层保留自愈逻辑

### 问题 3：日志收集
**选项 A**：Kestra 内置日志（推荐）
**选项 B**：接入外部日志系统

**建议**：选项 A，简化架构，Kestra UI 已足够

## 共识确认

1. 使用 Kestra 作为唯一的任务调度中心
2. 所有 pipeline 脚本通过 Kestra 工作流调用
3. 保留本地开发调试能力（直接运行脚本）
4. 失败时发送 Slack 通知（已配置 webhook）
5. 工作流参数通过 Kestra inputs 传递
