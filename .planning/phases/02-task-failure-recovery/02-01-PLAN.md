## Phase Goal

**As a** system operator, **I want to** have failed tasks automatically retry with exponential backoff and receive DingTalk alerts when tasks persistently fail, **so that** transient failures self-heal and I am notified of real problems requiring investigation.

<objectjective>
Add per-task retry with exponential backoff to the scheduler engine's job callback and a lightweight DingTalk alert notifier for persistent failure notifications.

Purpose: Transient failures (network blips, temporary API limits) self-heal without human intervention. Only genuine persistent failures trigger alerts, reducing noise and operator fatigue.

Output: Retry loop in engine.py, AlertNotifier in notifier.py, retry_count in history, 5+ passing tests.
</objective>

Modify engine.py to add retry logic inside the job callback's inner try block (after lock acquired, after record_start). The retry loop replaces the current single run_subprocess call with a while loop that respects self._scheduler_config.max_retries.

In __init__, add self._notifier = AlertNotifier() after existing resilience module initialization. Import AlertNotifier from services.data_service.scheduler.notifier.

In _create_job_callback, restructure the inner try block (lines 148-203 in current code) into a retry loop:

1. Initialize retry_count = 0, last_error = "", last_returncode = -1 before the loop.
2. Loop while retry_count <= max_retries (from self._scheduler_config.max_retries).
3. On each iteration: if retry_count > 0, calculate backoff delay as min(retry_delay * retry_backoff^(retry_count-1), max_backoff) using values from self._scheduler_config, log the retry attempt, and call time.sleep(backoff_delay).
4. Execute run_subprocess as before.
5. If returncode == 0: record success with retry_count, call circuit_breaker.record_success, break out of loop (success).
6. If returncode != 0: store stderr as last_error, store returncode as last_returncode, increment retry_count, continue loop.
7. Catch subprocess.TimeoutExpired separately (per D-01: timeout counts as a failed attempt) -- increment retry_count, set last_error to timeout message, continue loop. The timeout is already handled by run_subprocess raising TimeoutExpired.
8. After the loop exits (either success or retries exhausted): if the loop exited without success (retry_count > max_retries), record failure with retry_count=retry_count-1 (the actual number of retries, not attempts), call circuit_breaker.record_failure, and if task.alert_on_failure is True, call self._notifier.send_failure_alert(task.name, last_error, retry_count-1, max_retries).

The outer try/except (crash isolation) remains unchanged. The finally block (lock release) remains unchanged.

Key constraints per CONTEXT.md decisions:
- Retry happens WITHIN a single job callback (loop), not as separate APScheduler jobs (per D-06 and CONTEXT.md specifics)
- Config values come from SchedulerConfig global defaults (max_retries=3, retry_delay=60, retry_backoff=2, max_backoff=600) -- no per-task retry override needed in this phase
- Successful retry calls circuit_breaker.record_success, preventing false alarms (per D-04)
