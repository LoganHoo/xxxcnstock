## Phase Goal

**As a** system operator, **I want to** start the scheduler and have it run all configured tasks continuously without crashing, **so that** daily pipeline jobs execute on schedule without manual intervention.

## Acceptance Criteria

- [ ] Scheduler starts, loads config/scheduler.yaml, and registers all tasks
- [ ] When a task script crashes (non-zero exit), scheduler stays alive and logs the error
- [ ] When a task times out, subprocess is force-killed (no zombie processes)
- [ ] SIGTERM triggers graceful shutdown (stops accepting jobs, waits for running jobs)
- [ ] Health endpoint at /health returns `{"status": "alive"}` with uptime

<objective>
Walking Skeleton: Build the thinnest end-to-end scheduler that starts, runs tasks via subprocess, survives task crashes, and shuts down cleanly.

Purpose: Prove the core stability value proposition -- scheduler process stays alive no matter what individual tasks do.
Output: A runnable scheduler engine process with config loading, subprocess execution, crash isolation, and signal handling.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| config file -> scheduler | YAML config is read from filesystem; if compromised, could inject malicious script paths |
| scheduler -> subprocess | Scheduler spawns child processes; script paths must be validated |
| health endpoint -> network | HTTP endpoint exposes scheduler status on configurable port |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | config.py | mitigate | Validate script paths exist and are within project root before registering jobs. Use yaml.SafeLoader (not yaml.Loader) to prevent deserialization attacks. |
| T-01-02 | Tampering | executor.py | mitigate | Resolve script paths against project root, reject paths containing '..' components. Never pass unvalidated strings to subprocess. |
| T-01-03 | Information Disclosure | health endpoint | accept | Health endpoint returns only status/uptime/task count -- no sensitive data. Internal network only, no auth needed. |
| T-01-04 | Denial of Service | ThreadPoolExecutor | mitigate | Set explicit max_workers (default 4) and max_instances=1 per job to prevent thread exhaustion per RESEARCH.md Pitfall 1. |
</threat_model>
