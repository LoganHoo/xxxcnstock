---

## 问题陈述

### 现状问题

当前定时任务系统存在以下问题：

1. **平台依赖**: macOS LaunchAgent 仅支持 macOS，无法在 Linux 服务器部署
2. **配置分散**: cron_tasks.yaml 和 scheduler.yaml 两套配置，职责不清
3. **状态不可见**: 任务执行状态、失败原因缺乏统一监控
4. **分布式锁缺失**: 多实例部署时存在任务冲突风险
5. ** watchdog 未运行**: LaunchAgent 未安装，定时任务实际未启动

### 业务影响

- 数据采集任务无法按计划执行
- 缺乏自动化运维能力
- 无法在 Docker 环境中统一管理

---

## 目标

将 macOS LaunchAgent 定时任务迁移到 **Docker APScheduler**，实现：

1. **跨平台统一**: macOS/Linux/Docker 环境一致
2. **配置集中化**: 统一管理所有定时任务
3. **分布式锁**: Redis 实现多实例协调
4. **健康监控**: 任务执行状态实时可见
5. **优雅重启**: 无 downtime 部署更新

---

## 预期收益

| 指标 | 现状 | 目标 | 提升 |
|-----|------|------|------|
| 跨平台支持 | 仅 macOS | 全平台 | +100% |
| 任务可观测性 | 无 | 日志+指标 | 新增 |
| 多实例支持 | ❌ | ✅ Redis锁 | 新增 |
| 部署方式 | 手动 | Docker Compose | 自动化 |

---

## 范围

### 包含

- Docker APScheduler 服务设计
- 统一配置文件 (scheduler.yaml)
- Redis 分布式锁实现
- 任务健康检查机制
- 日志与监控集成

### 不包含

- 具体的业务任务脚本修改
- Kestra 工作流集成（本项目已有独立方案）
- 实时交易信号系统

---

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|-----|--------|------|---------|
| Redis 单点故障 | 低 | 高 | 配置 Redis Sentinel |
| 时区问题 | 中 | 中 | 统一 Asia/Shanghai |
| 任务执行超时 | 中 | 中 | 超时中断+告警 |

---

## 备选方案

### 方案A: Docker APScheduler（推荐）

- 优点: 轻量、跨平台、与现有 Docker Compose 集成
- 缺点: 需要维护独立容器

### 方案B: Kestra

- 优点: 强大的工作流编排能力
- 缺点: 引入新组件，学习成本高

### 方案C: 保留 LaunchAgent

- 优点: 无需改动
- 缺点: 无法跨平台，macOS 专属
