## Context

### 当前状态

XCNStock 数据流水线目前采用串行处理方式，每日收盘后执行以下流程：

1. **数据采集**: 遍历 5000+ 只股票，串行调用 Tushare API 获取日 K 线数据
2. **数据存储**: 将数据写入 Parquet 文件（每只股票一个文件）
3. **因子计算**: 使用 Pandas 计算技术指标和量价因子
4. **选股扫描**: 应用过滤器筛选符合条件的股票
5. **报告生成**: 输出选股报告

**当前性能瓶颈**:
- 数据采集: ~25 分钟（串行 HTTP 请求）
- 因子计算: ~8 分钟（Pandas 单线程处理）
- 总耗时: ~35-40 分钟

### 技术约束

- **Tushare API 限制**: 每分钟 500 次调用，需要流量控制
- **内存限制**: Docker 容器内存限制 4GB
- **数据一致性**: 必须确保 Kestra 和 APScheduler 不会重复执行同一任务
- **回滚能力**: 优化过程中不能破坏现有数据格式和接口

### 利益相关者

- **量化研究员**: 需要及时获取选股结果（20:00 前）
- **运维人员**: 需要监控任务执行状态
- **开发团队**: 需要可维护的代码架构

## Goals / Non-Goals

**Goals:**
- 将数据采集时间从 25 分钟缩短至 < 10 分钟（减少 60%）
- 将因子计算时间从 8 分钟缩短至 < 3 分钟（减少 60%）
- 实现 Kestra 与 APScheduler 的可靠状态同步
- 添加任务执行监控和告警机制
- 保持现有 API 接口向后兼容

**Non-Goals:**
- 不引入新的数据源（继续使用 Tushare/AKShare）
- 不修改 Parquet 文件格式（保持兼容性）
- 不重构整个架构（在现有基础上优化）
- 不实现实时数据流（仍使用日级批量处理）

## Decisions

### 1. 并行数据采集策略

**决策**: 使用 `aiohttp` + `asyncio` 实现异步批量采集，配合信号量控制并发数

**理由**:
- `aiohttp` 支持连接池复用，减少 TCP 握手开销
- 异步 I/O 可在等待 API 响应时处理其他请求
- 信号量可控制并发数，避免触发 Tushare 限流

**替代方案**:
- `multiprocessing`: 进程开销大，不适合 I/O 密集型任务
- `threading`: GIL 限制，无法充分利用多核
- `requests-futures`: 线程池模型，不如 asyncio 轻量

**实现细节**:
```python
# 并发控制：最多 50 个并发请求
semaphore = asyncio.Semaphore(50)
# 批量大小：每批 100 只股票
batch_size = 100
# 重试策略：失败重试 3 次，指数退避
retry_policy = ExponentialBackoff(max_retries=3)
```

### 2. 增量数据处理机制

**决策**: 基于 `trade_date` 和 `code` 的哈希值判断数据是否需要更新

**理由**:
- A 股数据一旦收盘即固定，无需重复下载历史数据
- 只需采集缺失日期的数据，大幅减少 API 调用
- 哈希校验可检测数据变更（如复权调整）

**实现细节**:
```python
# 检查本地最新日期
local_latest = get_latest_trade_date(code)
# 只采集缺失日期
if target_date > local_latest:
    fetch_data(code, target_date)
```

### 3. Polars 全面替代 Pandas

**决策**: 因子计算全面迁移至 Polars，仅在必要时与 Pandas 互操作

**理由**:
- Polars 查询性能比 Pandas 快 10-100 倍
- 原生支持多线程，自动利用多核 CPU
- 内存效率更高，Lazy API 支持查询优化

**替代方案**:
- `modin`: 分布式 Pandas，引入复杂度高
- `vaex`: 适合大数据集，但生态不如 Polars 成熟

**迁移策略**:
- 新代码直接使用 Polars
- 现有因子逐步迁移，保持接口兼容
- 使用 `to_pandas()` / `from_pandas()` 进行桥接

### 4. 多级缓存架构

**决策**: L1 内存缓存（`cachetools.LRUCache`）+ L2 Redis 缓存

**理由**:
- 内存缓存访问速度 < 1μs，适合热点数据
- Redis 提供进程间共享和持久化能力
- 两级缓存命中率预计 > 90%

**缓存策略**:
| 数据类型 | L1 内存 | L2 Redis | TTL |
|---------|---------|----------|-----|
| 股票列表 | ✓ | ✓ | 1 天 |
| 行业分类 | ✓ | ✓ | 7 天 |
| K 线数据 | ✗ | ✓ | 永久 |
| 因子结果 | ✓ | ✗ | 1 小时 |

### 5. 双调度器协调方案

**决策**: 基于 Redis 分布式锁 + 任务状态表实现互斥执行

**理由**:
- Redis 单线程特性天然支持原子操作
- 分布式锁可确保同一任务只有一个实例运行
- 状态表记录任务执行历史，便于故障排查

**状态流转**:
```
PENDING → RUNNING → SUCCESS/FAILURE
   ↑___________|
   (超时重试)
```

**锁机制**:
- 锁超时：任务最大执行时间的 2 倍
- 锁续期：任务执行中定期续期（heartbeat）
- 死锁检测：超过超时时间自动释放

### 6. Parquet 分区策略

**决策**: 保持单股票单文件，但添加日期分区目录

**理由**:
- 现有代码依赖单股票文件结构，改动风险高
- 日期分区便于按时间范围查询
- 与 Polars 的 `scan_parquet` 配合良好

**目录结构**:
```
data/kline/
├── 2024/
│   ├── 01/
│   │   ├── 000001.SZ.parquet
│   │   └── 600000.SH.parquet
│   └── 02/
└── 2025/
```

## Risks / Trade-offs

### 风险 1: 异步代码复杂度
**风险**: 异步代码调试困难，可能出现竞态条件  
**缓解措施**:
- 使用 `asyncio.gather()` 批量管理任务
- 添加详细的日志记录
- 提供同步包装函数作为 fallback

### 风险 2: 内存使用激增
**风险**: 并行处理可能导致内存溢出  
**缓解措施**:
- 限制并发数（信号量控制）
- 使用生成器流式处理数据
- 添加内存监控和告警

### 风险 3: 缓存一致性问题
**风险**: 缓存数据与源数据不一致  
**缓解措施**:
- 缓存设置合理的 TTL
- 提供手动刷新缓存接口
- 关键操作绕过缓存直接查询源

### 风险 4: 分布式锁失效
**风险**: Redis 故障导致锁机制失效  
**缓解措施**:
- 锁设置超时时间，防止死锁
- Redis 使用主从复制提高可用性
- 添加锁状态监控

### 权衡 1: 代码复杂度 vs 性能
- 接受异步代码和缓存逻辑带来的复杂度
- 收益：性能提升 60%+

### 权衡 2: 内存使用 vs 速度
- 接受更高的内存占用（缓存 + 并行）
- 收益：响应速度显著提升
- 缓解：Docker 内存限制调整为 8GB
